# Google Maps / Navigation System -- Simplified Interview Version

This is the whiteboard-friendly version. Use this in a 45-minute interview where the interviewer wants breadth over implementation depth.

---

## 1. Problem Statement (30s)
We need a mapping and navigation service that serves map tiles to billions of requests daily, computes optimal routes on a road graph with 500M intersections, and incorporates real-time traffic data. The hard part is making route computation fast enough -- plain Dijkstra on 500M nodes takes 5-10 seconds -- while keeping traffic data fresh within 2 minutes.

## 2. Requirements
**Functional:** Serve map tiles at multiple zoom levels. Compute optimal routes between origin and destination. Estimate travel time with real-time traffic. Place search and geocoding. Turn-by-turn navigation with re-routing on deviation.

**Non-functional:** Tile serving under 100ms p99. Routing under 500ms for routes up to 500km. 99.99% tile serving availability. Traffic data reflected within 2 minutes. ETA accuracy within 10% of actual.

## 3. Core Concept: Contraction Hierarchies
Plain Dijkstra explores millions of nodes per query. Contraction Hierarchies (CH) precompute the graph by iteratively removing less important nodes (local roads first) and adding shortcut edges that preserve shortest-path distances. Important nodes (highway junctions) float to the top. At query time, bidirectional search only goes "upward" in the hierarchy, exploring just 1,000-5,000 nodes instead of millions, completing in 1-10ms. For live traffic, Customizable CH (CCH) updates shortcut weights every 1-2 minutes without recomputing the entire structure.

## 4. High-Level Architecture
```
Client --> [CDN] --> [Tile Server] -- pre-rendered vector tiles from S3
                                       (99%+ cache hit rate)

Client --> [Routing Engine] -- Contraction Hierarchies in memory (~60GB)
                |
         [Traffic Pipeline] -- GPS probes --> map matching --> segment speeds
                |
         [Redis] -- live segment speeds updated every 1-2 min
```
- **Tile Server + CDN**: Pre-rendered vector tiles cached at CDN with 7-day TTL and 99%+ cache hit rate.
- **Routing Engine**: Contraction Hierarchies loaded in memory. 1-10ms per route query before traffic adjustment.
- **Traffic Pipeline**: Map-matches 10M GPS probes/minute to road segments using HMM, computes live speeds.
- **ETA Service**: Sums segment_length / predicted_speed across the route, accounting for time-of-arrival at each segment.

## 5. How Route Computation Works
1. User requests route from origin to destination.
2. Routing Engine runs bidirectional CH search: forward from origin going "up" the hierarchy, backward from destination going "up."
3. Both searches meet at a high-level node (typically a highway junction).
4. Shortcut edges are unpacked to recover the actual road-level path.
5. Edge weights incorporate live traffic speeds from Redis (updated every 1-2 minutes via CCH).
6. ETA is computed per-segment considering predicted traffic at time-of-arrival.
7. Alternative routes are generated by penalizing edges of the primary route (2x weight) and re-running.

## 6. What Happens When Things Fail?
- **Traffic pipeline lag**: Fall back to historical traffic patterns (speeds by time-of-week). Flag ETA as "estimated without live traffic."
- **Routing engine crash**: Stateless replicas reload the graph from shared storage (2-3 min warmup). Multiple replicas ensure availability.
- **CDN cache miss storm**: Origin tile server behind auto-scaling group with request deduplication for identical tiles.

## 7. Scaling
- **10x**: Regional routing shards (partition graph by continent/country). Cross-region routing uses a highway-only graph. More CDN PoPs for tiles.
- **100x**: GPU-accelerated parallel routing for batch queries. Predictive tile pre-fetching based on viewport movement. Compressed graph reduces memory from 50GB to 15GB. Federated routing with border handoff between countries.

## 8. Key Trade-offs to Mention
| Decision | Trade-off |
|----------|-----------|
| Contraction Hierarchies vs A* | CH is 100-1000x faster for long routes but requires hours of precomputation; A* is better for very short routes |
| Vector tiles vs raster tiles | Vector tiles are 80% smaller and style-flexible but require client-side rendering capability |
| HMM map matching vs nearest-segment | HMM handles noise and ambiguity much better, especially at complex intersections |

## 9. Closing (30s)
> Three pillars: CDN-served vector tiles with 99%+ cache hit rate for sub-100ms rendering, Contraction Hierarchies for sub-10ms routing on a 500M-node road graph, and a real-time traffic pipeline that map-matches 10M GPS probes per minute to compute live segment speeds. Customizable CH updates edge weights every 1-2 minutes without full recomputation. The system scales through regional graph sharding and hierarchical routing for cross-region trips.
